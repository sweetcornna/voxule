# voxlue 安静像素 Agent 设计访谈记录

> 日期：2026-05-28  
> 状态：进行中  
> 目标：通过逐轮追问，明确 voxlue 中像素宠物 agent 的产品定位、形象、能力边界、推送语气与疗愈体验。

## 已知项目事实

- voxlue 是“声音胶囊”应用，核心闭环是录音、装裱、埋下、显影、回放。
- 当前工程已有 agent 闭环骨架：`SignalDistiller`、`StateDigest`、`AgentGateway`、`IntelligenceService`、`SurfacedCapsuleView`、cadence 设置、HealthKit 授权页、Live Activity/灵动岛相关源文件。
- 现有合规边界：原始 HealthKit 数据只在设备内；网络边界只允许抽象 `StateDigest` 越过。
- 现有产品措辞边界：定位是陪伴，不使用临床或医疗承诺；仓库已有临床措辞扫描脚本。
- 用户希望新增/深化一个 agent：形象是像素宠物，气质接近 Claude Code 吉祥物；功能是分析用户埋下的胶囊，侧写偏好、性格与当时心情，并通过灵动岛信息或语音推送浮现内容。
- 用户明确不希望它成为“即时交互的普通语言模型”，而是一个安静的倾听者。
- 用户希望 app 对自闭患者具有心理疗愈效果，但需要在产品文案和功能边界上避免医疗化表达。

## 访谈结论

### 1. 第一身份

用户选择：C. 像素宠物。

结论：这个 agent 的第一身份应是“像素宠物”，而不是医生、咨询师、普通聊天助理、纯工具或抽象状态指示器。它需要有可被用户感知和记住的生命感、陪伴感与弱拟人关系，但后续仍需限定它的交互边界，避免变成即时聊天机器人。

### 2. 用户关系

用户选择：A/B/C 都要具备。

结论：像素宠物应同时具备“共处型宠物”“轻养成型宠物”“守护型宠物”三层关系。优先级上，共处感是底层体验：它安静住在 app 里，不索取注意力；养成感应轻量存在，提供状态变化、熟悉感和长期陪伴感，但不设计强打卡、惩罚、饥饿或退化；守护感服务于胶囊机制，表现为守着用户埋下的声音、在合适时机轻轻提醒、帮用户把重要片段重新递回来。

### 3. 养成强度

用户选择：A. 纯视觉成长 + B. 情感熟悉度成长。

结论：养成系统不设计任务、喂食、连续登录、惩罚、数值退化或强游戏化循环。后续会为像素宠物设计不同性格对应的外观和动作；LLM/agent 的职责是分析用户埋下的胶囊与上下文，对用户偏好、性格倾向和当时心情做温和侧写，再判断应选取哪类外观、待机状态、小动作或浮现方式。LLM 不直接承担即时聊天，也不以“能力升级”诱导用户持续互动。

### 4. 侧写结果的呈现方式

用户选择：B. 间接呈现 + C. 半透明呈现。

结论：用户侧写默认不做显性的“人格报告”或“心情判定”。Agent 对偏好、性格倾向和当时心情的理解，主要通过像素宠物的外观、动作、语气、浮现节奏和推送时机间接体现；必要时可以用温和观察句半透明呈现，例如“你最近常把雨声留下来”“它好像记住了你喜欢安静的傍晚”。禁止输出“你是某某人格”“你现在焦虑/抑郁”“系统判断你需要被干预”等定义式、诊断式或压迫感强的表达。

### 5. 心情与偏好分析的输入来源

用户选择：主要使用 A. 胶囊内容线索 + B. 声音线索；C. 设备/健康粗信号作为后续功能创新扩展。

结论：Agent 的主要分析依据应来自胶囊本身和声音本身，包括标题、标签、地点名、时间、锁类型、是否回听、音量、停顿、环境声类型、是否有人声等。设备/健康粗信号是有价值的创新方向，但应放入后续扩展，不作为首要依赖；加入时必须保持现有隐私边界，即原始健康数据不出设备，只允许抽象、粗粒度摘要参与决策。

### 6. 宠物模式与声音特征方向

用户补充：用户在一开始选择/领养 agent 宠物时，可以选择两种模式：一种更安静，一种更贴心。Agent 能通过用户语音特征进行分析，技术方向可参考开源 TTS 和 GPT-SoVITS。

结论：Agent 应在首启或领养流程中提供“更安静”和“更贴心”两种陪伴模式。两者不是不同产品，而是同一像素宠物的不同陪伴密度：安静模式减少主动出现和主动表达，贴心模式允许更主动的观察句、语音触达和状态回应。声音特征分析是后续体验核心之一，可用于判断宠物外观、动作、浮现语气和触达方式；开源 TTS/GPT-SoVITS 可作为语音生成或声音风格实验方向，但仍需继续明确原始声音是否允许离开设备、是否允许训练/克隆用户声音，以及用户授权边界。

### 7. 原始音频与深度语音分析边界

用户选择：C. 默认端侧摘要；用户明确授权后，才做更深的云端或本地模型分析。

结论：默认情况下，原始音频不应直接进入云端 agent；端侧先提取声音特征摘要，供 agent 做外观、动作、语气和浮现判断。若未来需要使用更深的语音模型能力，例如云端音频理解、TTS 风格适配、GPT-SoVITS 相关音色实验或更细粒度语音特征分析，必须在用户明确授权后启用，并需要把“分析用途、是否上传、是否留存、是否用于训练/个性化”分开说明和控制。

### 8. 贴心模式的主动程度

用户选择：B. 温和主动：可灵动岛/通知/短语音，但不自由聊天。

结论：贴心模式允许 agent 更主动地通过灵动岛、通知、宠物动作、短观察句和短语音进行触达，但不开放自由聊天入口，不做连续对话型普通语言模型。用户点开后应进入有限动作：听听看、等等再说、这次不准、更少出现、调整模式等，而不是进入聊天窗口。贴心模式的核心是更高陪伴密度，不是更强对话能力。

### 9. 安静模式的主动程度

用户选择：B. 低频主动：只在胶囊浮现/重要时机提醒。

结论：安静模式不是完全静默，也不是纯装饰。它保留守护型宠物的最低限度主动性：只有当某枚胶囊真正到了值得浮现的时机，才通过灵动岛或极短通知出现。平时主要通过 app 内待机动作和轻微状态变化表达存在感，不主动发起语音，不频繁写观察句，不制造“必须回应”的压力。

### 10. 语音推送形态

用户选择：A 为主，C 可作为贴心模式增强。

结论：语音推送应主要播放用户自己过去埋下的声音胶囊，这是 voxlue 的核心价值，也是最具情感真实性的部分。贴心模式可以在播放前加入 agent 的极短 TTS 引导句，再播放胶囊；agent 自己的声音不能抢过用户声音，不能长篇解释、分析或替用户定义情绪。安静模式默认只推送胶囊本身或极短无声提示。

### 11. 形象设计责任边界

用户说明：像素宠物的具体形象设计暂时待定，因为美术设计不由用户负责。

结论：当前设计访谈不再决策最终轮廓、物种、像素造型或具体美术风格。产品侧仍需要定义美术交付所依赖的行为约束：它是像素宠物，有安静/贴心两种陪伴密度，需要能表达共处、轻养成、守护、浮现胶囊等状态；最终形象由美术侧在这些行为与品牌约束内探索。后续访谈重点转向 agent 的状态系统、触达规则、分析输出格式、用户控制权和安全边界。

### 12. MVP 行为状态

用户选择：A. 最小 5 状态；后续再添加更多行为。

结论：MVP 阶段只要求像素宠物支持 5 个基础行为状态：待机、聆听/记录后、守护中、带来胶囊、被用户婉拒后。暂不设计完整情绪状态集、复杂组合状态或大量动作库。后续可以扩展更多表情、动作、情绪外观和功能状态，但首版必须保证这 5 个状态足够清晰、可被美术实现、可被工程映射、可支撑核心胶囊浮现体验。

### 13. 状态文案规则

用户选择：B. 只在记录后、带来胶囊时说话。

结论：MVP 5 状态中，agent 只在“聆听/记录后”和“带来胶囊”两个节点出现短文案。待机和守护中主要用动作、姿态、图标或微动效表达存在感，不主动说话；被用户婉拒后也应尽量使用动作反馈，最多保留极短、无压力的确认句。这个规则服务于“安静倾听者”定位，避免每个状态都输出文本造成打扰。

### 14. 主动建议边界

用户选择：B. 只能建议听某段已有胶囊。

结论：Agent 不提供现实行动建议、生活建议、心理建议或干预建议。它的主动建议边界仅限 app 内的核心动作：递回某段已经存在的声音胶囊，并温和建议用户“听听看”。它不能说“你应该出去走走”“你需要联系某人”“做某个练习会好一点”等话。这样可以保持产品聚焦声音胶囊，也降低医疗化和过度引导风险。

### 15. Agent 分析输出格式

用户选择：B. 输出状态 + 动作 key + 胶囊 id + 一句短文案。

结论：LLM/agent 的一次分析结果应是结构化输出，而不是直接输出大段自然语言。MVP 输出字段应包括：像素宠物状态、外观/动作 key、是否浮现胶囊、胶囊 id、短文案，以及必要的模式差异参数。前端根据这些结构化字段渲染宠物动作、灵动岛/通知内容和浮现卡，不允许模型自由发挥成长篇聊天或完整人格报告。这样更稳定、可测试，也更符合“非即时聊天模型”的定位。

### 16. 短文案初始语气

用户选择：B. 像素宠物；初始限定为该语气。

结论：MVP 短文案的初始语气限定为“像素宠物在用很少的话表达观察”。它不是系统通知、不是人类朋友、不是暗房冲洗师，也不是心理咨询师。语气应有轻微生命感，但句子必须短、含蓄、低压力，不解释用户、不定义用户、不进行心理判断。暗房/胶片隐喻可保留在视觉和页面氛围里，但不作为 agent 初始文案主声线。

### 17. 文案人称规则

用户选择：C. 默认“它”，贴心模式少量“我”。

结论：Agent 文案默认使用第三人称或事件描述，例如“它把这段声音带来了”“这段声音浮上来了”，以降低拟人压力和被注视感。在贴心模式、极短 TTS 或记录后反馈中，可以少量使用第一人称，例如“我听见了”“我把它收好了”。第一人称必须克制，不用于长篇表达、情绪判断或现实建议。

### 18. MVP 用户控制权

用户选择：A. 安静/贴心模式切换 + B. 对某次浮现说“这次不准/别推这类” + C. 调低出现频率或暂停一段时间。

结论：MVP 必须给用户清晰的控制权：可以在安静/贴心模式之间切换；可以对单次浮现表达拒绝或反馈“别推这类”；可以调低 agent 出现频率或暂停一段时间。关闭声音特征分析、清除 agent 记住的偏好/画像也重要，但可作为后续隐私设置或完整隐私面板能力规划，不强制进入 MVP。

### 19. 负反馈学习方式

用户选择：B. 降低相似胶囊/相似时机权重，不追问。

结论：当用户选择“这次不准/别推这类”后，agent 应静默学习，而不是追问原因或弹出问卷。系统只需要降低相似胶囊、相似标签、相似声音特征或相似时机的未来浮现权重。拒绝后的即时反馈应短而无压力，例如动作上退回、收好胶囊；不要求用户解释，也不表达失落或内疚感。

### 20. 首启领养流程

用户选择：B. 只选“更安静 / 更贴心”，不问性格问卷。

结论：用户一开始领养 agent 时，只需要选择陪伴模式：更安静或更贴心。MVP 不做性格问卷、不要求用户选择偏好、不让用户填写声音类型或提醒时间等复杂配置。其他偏好由后续胶囊、回听行为和拒绝反馈慢慢学习。这可以降低进入成本，也符合“它慢慢认识你”的陪伴感。

### 21. 长期记忆范围

用户选择：B. 只保存轻量偏好权重，不保存可读画像文本。

结论：Agent 的长期记忆在 MVP 中只保存轻量偏好权重，例如雨声、傍晚、低频提醒、某类胶囊被拒绝等权重。系统不保存“用户是内向敏感型”“用户偏孤独”“用户近期情绪低落”这类可读画像文本，避免标签化、误判和隐私风险。Agent 的“慢慢认识你”应体现为浮现选择、动作和语气逐渐更合适，而不是形成一份可读的人格报告。

### 22. 长期记忆存储边界

用户选择：A. 全部本地存储，云端只参与单次推理。

结论：MVP 中轻量偏好权重全部保存在本地，例如 SwiftData 或 UserDefaults。云端 agent 只参与单次推理，每次只接收必要摘要，不保存用户级长期记忆，不沉淀可回溯画像，也不把用户声音/心理敏感相关数据放进自建云端长期存储。这个边界更适合声音与心理敏感场景，也与现有 `StateDigest` 的隐私设计方向一致。

### 23. 云端单次推理输入上限

用户选择：B. 胶囊元数据 + 声音特征摘要 + 本地偏好权重摘要。

结论：云端单次推理最多接收胶囊元数据、端侧提取的声音特征摘要、本地偏好权重摘要。不上传原始音频，不上传完整历史，不上传可读画像文本。云端推理的核心目标是根据这些摘要推断用户当下的心理状态或心绪倾向，为后续推送、浮现时机、宠物动作和短文案提供依据；但这些推断不应直接以诊断式标签展示给用户，也不应被云端长期保存。

### 24. 内部分析粒度

用户选择：B. 低维心绪状态 + C. 细分心理画像，但分析结果不对用户展示。

结论：Agent 内部可以同时进行两层分析：一层是低维心绪状态，例如安静、低能量、怀旧、紧绷、社交温暖等；另一层是更细的性格/偏好倾向，例如声音偏好、社交能量、对提醒频率的接受度、对某类胶囊的回避或亲近。两层结果都只作为推送决策依据，不直接展示给用户，不生成可读画像报告，不使用诊断式或人格定型标签。用户实际收到的只是被选择出的声音胶囊、宠物动作和极短提示。产品目标可以表述为“在非正面心绪中递回可能有帮助的声音”，但不应承诺改变、治疗或纠正用户的心情状态。

### 25. 非正面心绪下的推送策略

用户选择：B. 推稳定胶囊：熟悉、低刺激、安心、可预测的声音。

结论：当 agent 内部判断用户可能处于非正面心绪时，推送策略应优先选择稳定胶囊，而不是强行积极、热闹或鼓励类内容。稳定胶囊的特征包括熟悉、低刺激、安心、可预测，例如雨声、房间声、安静地点声、稳定重复的环境声等。目标是轻轻递回一个可能让用户停靠的声音片段，而不是要求用户立刻变得积极。

### 26. 推送时机判断

用户选择：A. 固定时间 + C. 心绪窗口、胶囊适配、打扰成本共同判断。

结论：Agent 的推送不应完全随机，也不应只靠单一触发条件。MVP 可以设置固定检查窗口，例如每天或每周在低打扰时段运行一次 agent 判断；但固定时间只是“检查时机”，不是“必然推送”。真正发出灵动岛、通知或语音触达前，仍需同时满足三类条件：用户可能处于适合接收稳定声音的心绪窗口、有匹配的稳定胶囊可递回、当前打扰成本较低。这样既有节奏感，也能避免机械打扰。

### 27. 固定检查频率

用户选择：B. 安静模式每周 2-3 次，贴心模式每天 1 次。

结论：Agent 的默认检查频率按陪伴模式区分：安静模式每周检查 2-3 次，贴心模式每天检查 1 次。检查不等于推送；每次检查仍必须通过心绪窗口、胶囊适配和打扰成本判断，只有三者共同满足时才触达用户。这个频率设计用于提供稳定陪伴节奏，同时避免把 agent 变成高频打扰源。

### 28. 连续忽略后的退让机制

用户选择：B. 自动降频，不表达失落。

结论：如果用户连续几次没有点开推送、选择“等等再说”，或对浮现保持低响应，agent 应自动降低出现频率，而不是继续按原频率触达。降频过程应安静发生，不弹出追问，不表达失落、不委屈、不责备用户，也不使用“你很久没理我了”这类情绪勒索式文案。低压力退让是像素宠物陪伴感的一部分。

### 29. 没有合适胶囊时的行为

用户选择：A. 不推送，保持沉默。

结论：如果 agent 判断当前没有合适胶囊，它应保持沉默，不发送无胶囊问候、不建议用户立刻埋下一段新声音，也不为了完成推送而选择低匹配内容。像素宠物可以在 app 内维持“守护中”状态，但不主动触达用户。克制是这个 agent 的核心体验原则之一。

### 30. 强烈负面状态下的安全兜底

用户选择：B. 避免刺激性推送，保持低刺激，不做危机干预。

结论：如果胶囊内容或语音特征暗示用户可能处于很强烈的负面状态，agent 在 MVP 中不承诺识别危机、不做危机干预、不弹出强提示，也不把判断展示给用户。它应做的事是避免推送可能刺激、热闹、强情绪或高唤醒的内容，优先保持低刺激、熟悉、可预测；如果没有合适的稳定胶囊，就保持沉默。

### 31. MVP 明确不做的能力

用户选择：都不做。

结论：MVP 明确排除以下能力：自由聊天、用户画像报告页、原始音频云端上传、声音克隆/GPT-SoVITS 个性化音色、危机识别或危机干预。首版 agent 应聚焦“结构化分析 + 像素宠物状态 + 胶囊浮现 + 低压力控制权”，不进入普通聊天机器人、心理画像产品、语音克隆产品或严肃安全干预系统。

### 32. MVP 成功标准

用户选择：它懂得什么时候安静，什么时候递来一段合适的声音。

结论：首版 agent 的核心成功标准不是聊天时长、推送数量或高频活跃，而是用户感到它“懂得什么时候安静，什么时候递来一段合适的声音”。这意味着它需要同时满足两件事：不合适时保持沉默，合适时能选择一段低压力、稳定、贴合当下的声音胶囊。后续验证和指标设计都应围绕“合适、不打扰、愿意保留”展开。

### 33. 合适性的验证方式

用户选择：B. 回听完成/停留 + C. 负反馈率。

结论：MVP 判断“递来的声音是否合适”时，不以单纯点开率为核心，也不要求用户额外打分。主要观察两个低摩擦信号：用户是否真的听了一段被浮现的胶囊，以及用户是否点击“这次不准/别推这类”等负反馈。合适的推送应表现为较高回听/停留、较低负反馈；连续忽略或负反馈应进入自动降频和权重调整。

## 待澄清决策树

- Agent 的核心定位：宠物、守信人、冲洗师、陪伴者，或混合角色。
- 用户画像与敏感边界：是否直接面向自闭患者，还是更广义的高敏感/低社交能量人群。
- 分析深度：只做温和偏好归纳，还是形成长期人格/心情档案。
- 主动触达边界：什么时候推送、以何种频率、能否沉默。
- 形象系统：像素宠物的物种/轮廓、状态、动效、配色与现有暗房胶片美学的关系。
- 语言人格：少说、含蓄、准确，还是更拟人、更亲密。
- 语音策略：只读出用户自己的声音胶囊，还是 agent 也可以生成声音。
- 安全与合规：避免诊断、治疗承诺、过度情绪判断和依赖诱导。
